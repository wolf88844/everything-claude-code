# Topic Library Spec

Use this spec to keep the topic system consistent across:
- Notion topic databases
- `topic-seed-library.md`
- workflow outputs
- post-draft and post-publish reviews

The goal is not to force one storage tool.
The goal is to make every topic trackable with the same language.

## Core Topic Fields

Each topic should have these fields whenever possible:
- `Topic` -> the working idea or seed topic
- `Content pillar` -> which long-term pillar the topic belongs to
- `Publishing line` -> which strategic line it serves
- `Role` -> what job the topic plays in the content sequence
- `Status` -> where the topic currently sits in the workflow
- `Latest title` -> the strongest current title version
- `Mother theme` -> the deeper recurring theme behind the topic
- `Derived from` -> parent topic if this is an expansion or rewrite
- `Notes` -> short learning, risk, or next-step note

## Content Pillar Labels

Use one primary pillar:
- `成长卡点拆解`
- `认知升级与自我管理`
- `技术职场判断力`
- `经验变内容 / 个人品牌起步`

If needed, add one secondary pillar in notes instead of over-tagging.

## Publishing Line Labels

Use one main line:
- `普适成长焦虑线`
- `技术人成长停滞线`
- `经验资产化线`
- `判断力与表达力线`
- `第二增长曲线线`

Do not create new lines casually.
Only add one if multiple topics clearly need it.

## Topic Role Labels

Use one primary role:
- `引流题`
- `主文题`
- `承接题`
- `短内容题`

Use these meanings:
- `引流题` -> broad recognition, easier entry, higher click potential
- `主文题` -> core account-defining article, stronger diagnosis or worldview
- `承接题` -> helps deepen trust, explain method, or connect to a bigger series
- `短内容题` -> suitable for short post, 朋友圈, or small social repurposing

## Topic Status Labels

Use these statuses consistently:
- `待写` -> approved seed, not drafted yet
- `草稿` -> drafted in workflow, not yet published or validated
- `已写` -> completed article, not yet validated by publishing feedback
- `已验证` -> supported by real publishing feedback or clear performance evidence
- `已重写` -> rewritten from a weak sample or upgraded from an older angle

## Mother Theme Labels

Keep mother themes reusable and limited.
Examples:
- `旧路径失效`
- `隐性失效点`
- `经验没有升级成资产`
- `判断力比执行更值钱`
- `表达力决定可见度`
- `忙碌不等于前进`

If a topic cannot map to a mother theme, the topic may still be too shallow.

## Derived Topic Rules

When a topic comes from another topic, record:
- `Derived from`
- what changed
- why this derivative exists

Common derivative reasons:
- sharper diagnosis
- narrower technical scene
- stronger authority angle
- short-content adaptation
- weak-sample rewrite

## Notes Format

Keep notes short and useful.
Good note examples:
- `已有满意初稿，可作为主线题继续迭代`
- `标题有点击力，但正文还不够贴技术现场`
- `适合做引流，不适合当第一篇代表作`
- `发布后评论强，说明机制命名打中了`

Avoid vague notes like:
- `不错`
- `可以再看看`
- `后续优化`

## Workflow Mapping Rules

At different workflow stages, update the topic like this:

### After Step 1 to Step 3
- confirm `Content pillar`
- confirm `Publishing line`
- confirm `Role`
- keep status as `待写` unless drafting starts

### After first full draft
- set status to `草稿`
- update `Latest title`
- add a short note on the strongest mechanism or main risk

### After article is finalized but not yet validated
- set status to `已写`
- keep the best title
- note what the article is trying to win

### After real publishing feedback
- decide whether to mark as `已验证`
- capture what actually worked
- note whether the topic deserves expansion

### After a rewrite from a weak sample
- set status to `已重写`
- preserve the old topic in notes
- record what changed in the new angle

## Minimal Topic Record Example

- `Topic`: 为什么很多技术人不是没经验，而是这些经验一直停留在“做过”
- `Content pillar`: 技术职场判断力
- `Publishing line`: 经验资产化线
- `Role`: 主文题
- `Status`: 草稿
- `Latest title`: 做过不等于留下：为什么很多技术人的经验还没有变成资产
- `Mother theme`: 经验没有升级成资产
- `Derived from`: 离职同事.skill 新闻入口
- `Notes`: 强机制题，适合做代表作，不适合写成泛 AI 评论
